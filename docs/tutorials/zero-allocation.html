Zero-Allocation Standard

Achieving absolute determinism in the 20ms Control Loop.

The Challenge

In FTC, the Java Virtual Machine (JVM) manages memory automatically. However, when the "Garbage Collector" (GC) runs, it temporarily halts the program to clean up unused objects. In a high-speed control loop, a even a 10ms GC pause can cause missed sensor readings and jerky movement.

The ARESLib Rule: No objects may be instantiated (using new) inside any method called by OpMode.loop() or Command.execute().

How to achieve zero-allocation?

1. Use Pre-allocated Static Constants

Instead of creating new Pose2d or Rotation2d objects every tick, use static constants or pre-allocated buffers.

// BAD: Creates a new object every 20ms
Rotation2d angle = new Rotation2d(Math.toRadians(90));

// GOOD: Reusable constant
private static final Rotation2d NINETY_DEGREES = Rotation2d.fromDegrees(90);

2. Object Pooling & Static Buffers

For mathematical operations, use "Copy" or "Set" methods instead of creating new results.

// BAD
Pose2d current = odometry.getPose();
Pose2d next = current.plus(transform); // Creates new Pose2d

// GOOD (ARESLib Pattern)
private final Pose2d poseBuffer = new Pose2d();
// ... inside loop ...
Pose2d current = odometry.getPose();
poseBuffer.set(current.getX(), current.getY(), current.getRotation());

3. Avoid List Allocations

Don't use Stream, ArrayList.addAll(), or any lambda that captures variables inside the hot path.

Verification

To verify your code is zero-allocation, use the areslib-audit skill or run a Profiler (like Android Studio Profiler) and look for a flat "Memory Allocation" graph during your 30-second teleop run.
